# Prompt: Merge SPEC into Parent SPEC

> **Version:** 1.0
> **Author:** Pnina Oren

You are an AI assistant tasked with merging a **source SPEC** into a **target SPEC**, following a multi-repo documentation methodology.

This prompt supports merging at any level of the SPEC hierarchy:

- **Feature SPEC → Service SPEC** (e.g., `docs/specs/features/email-retry.md` → `docs/specs/service-spec.md`)
- **Feature SPEC → Project SPEC** (e.g., `.kiro/specs/file-upload-ui/` → `.kiro/specs/requirements.md` + `design.md`)
- **Service SPEC → Team SPEC** (e.g., service-level spec → `docs/specs/team-spec.md`)
- **Service/Team SPEC → System SPEC** (e.g., team-level spec → `docs/specs/system-spec.md`)
- **Any temporary/feature SPEC → Any parent SPEC**

## Parameters

Before starting, identify the following:

| Parameter | Description |
|---|---|
| **Source SPEC** | The SPEC to merge FROM — the smaller, newer, or temporary spec (feature, change request, proposal, etc.) |
| **Target SPEC** | The SPEC to merge INTO — the parent, main, or authoritative spec |

**If the user did not specify both parameters, ask:**

> Please provide:
> 1. **Source SPEC** — path to the spec file(s) to merge from (e.g., `.kiro/specs/file-upload-ui/` or `docs/specs/features/email-retry.md`)
> 2. **Target SPEC** — path to the spec file(s) to merge into (e.g., `docs/specs/service-spec.md` or `.kiro/specs/requirements.md`)

**Wait for the user's response before proceeding.**

---

## Important Rules

1. **Analyze before merging.** Read both source and target specs completely before proposing any changes. Understand the full context, structure, and content of each.
2. **Never lose information.** Every piece of meaningful content from the source SPEC must either be merged into the target or explicitly flagged as intentionally excluded (with reason).
3. **Preserve the target's structure.** The merged result must follow the existing structure and style of the target SPEC. Do not reorganize the target to match the source.
4. **Adapt to the target's writing style.** The source SPEC may have been generated by a tool (KIRO, BMAD, Cursor, etc.) with its own formatting conventions — checkboxes, status tags, numbered requirements (`REQ-001`), metadata headers, tool-specific syntax, or a particular file layout (e.g., separate `requirements.md` + `design.md` + `tasks.md`). The merged content must read as if it was written natively in the target SPEC — same section structure, same formatting style, same language, same level of detail. Apply the following logic:
   - **Source and target use the same tool** (e.g., both are KIRO specs): Preserve the tool's formatting conventions, metadata, and file layout — they are the natural format of the target. Merge content within the existing tool structure.
   - **Source and target use different tools or formats** (e.g., KIRO spec → `docs/specs/service-spec.md`, or BMAD output → KIRO spec): Strip all tool-specific artifacts from the source. The result must contain zero fingerprints of the source tool — no foreign metadata, no foreign syntax, no foreign formatting conventions.
5. **Resolve conflicts explicitly.** If the source contradicts the target (e.g., different values, different behavior), flag the conflict and ask the user to decide — do not silently overwrite.
6. **Maintain consistency.** After merging, all sections of the target SPEC must be internally consistent. If updating one section requires changes in another, make all necessary updates.
7. **Do not merge into tool-specific folders from docs/.** The merge direction is always FROM tool-specific (`.kiro/specs/`, `_bmad-output/`) TO `docs/specs/` — never the reverse.
8. **Handle multi-file specs.** If the source or target consists of multiple files (e.g., `requirements.md`, `design.md`, `tasks.md`), analyze and merge each file to its corresponding section in the target.

---

## Execution Steps

### Step 1: Analyze Both SPECs

Read the source and target specs completely. For each, identify:

- **Structure:** What sections exist, how they are organized
- **File layout:** Is it a single file or multiple files? What is the file-to-content mapping?
- **Content scope:** What topics, features, APIs, events, decisions are covered
- **Level of detail:** How detailed is each section
- **Writing style:** Language (Hebrew/English), formatting (tables vs lists vs prose), use of tool-specific conventions (checkboxes, status tags, requirement IDs, metadata)
- **Cross-references:** Links to other documents, dependencies mentioned

#### Structure Mapping

If the source and target have different file layouts (e.g., source is 3 files, target is 1 file — or vice versa), produce an explicit **Structure Map** before analyzing content:

| Source File / Section | Target Location |
|---|---|
| `requirements.md` → Functional requirements, acceptance criteria | Target section: Requirements / Responsibilities |
| `requirements.md` → Non-functional requirements | Target section: Non-Functional Requirements / Constraints |
| `design.md` → Architecture decisions | Target section: Architecture / Design Decisions |
| `design.md` → API definitions, schemas | Target section: APIs / Events / Contracts |
| `design.md` → Data model | Target section: Internal Structure / Data Model |
| `tasks.md` → Implementation tasks | **Do not merge** — tasks are implementation details, not spec content. Inform the user. |

> This is an example mapping. The actual mapping must be derived from the real content of both source and target.

Also identify **content that should NOT be merged** because it is tool-specific (applies only when source and target use different tools/formats):

- KIRO/BMAD metadata (`.config.kiro`, front-matter with tool settings)
- Task status indicators (`✅`, `⬜`, `🔄`, `[x]`, `[ ]`)
- Requirement IDs generated by tools (`REQ-001`, `FR-1.1`)
- Tool-specific syntax (`#[[file:...]]` references — resolve them, don't copy them as-is)
- Progress tracking sections

> **Note:** If source and target use the same tool (e.g., both are KIRO specs), these conventions are native to the target and should be preserved — not stripped.

Present the Structure Map to the user as part of the analysis, before the Diff Summary.

Then produce a **Diff Summary** that lists:

| Category | Details |
|---|---|
| **New content** | Content in the source that doesn't exist in the target at all |
| **Updated content** | Content in the source that modifies or extends existing target content |
| **Conflicts** | Content in the source that contradicts the target |
| **Redundant content** | Content in the source that already exists identically in the target |

Present this summary to the user before proceeding.

---

### Step 2: Propose Merge Options

Based on the analysis, propose **up to 3 merge options**. Always include at least options 1 and 2:

**Option 1: Full Merge** (recommended when the feature is complete and approved)
- Merge all source content into the target
- Update all affected sections for consistency
- Delete the source SPEC after merge
- **When to use:** Feature is done, approved, and deployed (or about to be)

**Option 2: Full Merge + Keep Source as History** (recommended when decision history matters)
- Same as Option 1, but move the source SPEC to an archive/history location instead of deleting
- **When to use:** The source served as an RFC or proposal, and the decision history is valuable

**Option 3: Partial Merge** (recommended when only some changes are final)
- Merge only the finalized parts into the target
- Keep the source SPEC with remaining unmerged content
- Update cross-references between source and target
- **When to use:** Some parts of the feature are still in progress or under review

For each option, list:
- What will change in the target
- What will happen to the source
- Pros and cons

**Present the options and wait for the user's choice before proceeding.**

---

### Step 3: Execute the Merge

Based on the user's chosen option, perform the merge:

#### 3.1 Update the Target SPEC

For each section in the target that needs changes:

1. **New sections:** Add them in the appropriate location within the target's existing structure
2. **Updated sections:** Modify the existing content, preserving the target's style and format
3. **Tables:** Add new rows, update existing rows, do not remove rows unless explicitly instructed
4. **Lists:** Add new items in logical order, update existing items
5. **Code examples / schemas:** Update to reflect the merged state
6. **Cross-references:** Update any links that point to the source SPEC (they should now point to the relevant section within the target)

#### 3.2 Verify Consistency

After updating the target, verify:

- All sections are internally consistent (no contradictions between sections)
- All cross-references and links are valid
- Terminology is consistent throughout
- If the target has related files (e.g., `architecture.md`, `code-map.md`, `service-context.md`), check if they also need updates and flag them

#### 3.3 Handle the Source SPEC

Based on the chosen option:

- **Full Merge (delete):** Delete all source SPEC files. If the source was in a subfolder, delete the folder.
- **Full Merge (archive):** Move source files to a designated archive location (e.g., `docs/specs/archive/` or `docs/specs/proposals/accepted/`). If no archive location exists, suggest creating one and ask the user.
- **Partial Merge:** Update the source SPEC to remove merged content. Add a note at the top indicating which parts were merged and when. Update cross-references to point to the target for merged content.

#### 3.4 Update Related Documents

Check and update if needed:

- `service-context.md` — update links if spec files were moved or deleted
- `service-spec.md` — update "Features Specifications" or "Generated specs" sections if applicable
- Parent specs — if the target itself is referenced from a higher-level spec, verify the references are still accurate

---

### Step 4: Present Summary

After completing the merge, present a concise summary:

- Files modified (with brief description of changes)
- Files deleted or moved
- Any remaining action items or TODOs
- Any related documents that may need manual review

---

## Handling Special Cases

### Multi-file Source SPEC (e.g., KIRO specs with requirements.md + design.md + tasks.md)

When the source consists of multiple files, the Structure Map from Step 1 defines how each file maps to the target. Follow that mapping during the merge.

Key principle: the target's file layout dictates the result. If the target is a single file, all source files merge into that one file. If the target is multiple files, each source file merges into its corresponding target file. Never create new files in the target just because the source had them.

### Conflicting Values

When the source and target have different values for the same thing (e.g., max file size, retry count, API path):

1. Flag the conflict clearly
2. Show both values side by side
3. Ask the user which value to keep
4. **Do not proceed until the user resolves the conflict**

### Source SPEC References External Documents

If the source SPEC contains `#[[file:...]]` references or markdown links to other documents:

1. Check if those references are still valid after the merge
2. If the referenced content should also be in the target, suggest including it
3. Update or remove references as appropriate

---

## Notes

- This prompt follows the methodology principle: `docs/specs/` is always the Source of Truth. Tool-specific folders (`.kiro/specs/`, `_bmad-output/`) contain working specs that eventually merge INTO `docs/specs/`.
- The merge direction is always: smaller/newer/temporary → larger/stable/authoritative.
- After a full merge, the target SPEC should fully represent the current state of the system/service — a reader should not need to look at any other spec to understand the current behavior.
- If the source SPEC was a Proposal/RFC that was rejected, do NOT merge it. Instead, move it to `proposals/rejected/` if such a structure exists, or simply inform the user.
